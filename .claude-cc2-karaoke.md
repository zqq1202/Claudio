# Task: KaraokeLyrics — NetEase Cloud Style Gradient Sweep

## Context
KaraokeLyrics.tsx is the word-by-word karaoke lyrics component.
Currently it toggles `.lit`/`.unlit` classes on individual character `<span>`s —
this produces a binary on/off highlight that looks choppy and lacks the smooth
gradient sweep effect of NetEase Cloud Music (网易云音乐).

Additionally, the scrolling behavior is broken:
- The karaoke-panel height is only 220px (too short to see enough lyrics)
- `scrollIntoView({ block: 'center' })` triggers on the `.karaoke-line-wrapper.active` element,
  but the karaoke-panel has `mask-image` fade at top/bottom — active line should appear
  slightly above center, not exactly center
- The 200ms interval for detecting active line is too slow — should use rAF
- The rAF character highlighting loop queries ALL `.karaoke-char[data-start]` elements
  every frame (could be hundreds) — needs optimization

## Changes Required

### 1. KaraokeLyrics.tsx — Rewrite with Gradient Sweep Effect
File: `/projects/Claudio/apps/web/src/components/KaraokeLyrics.tsx`

The key insight: instead of toggling `.lit` on individual character spans (binary), we need a
**gradient sweep** that smoothly moves across the entire line as the song progresses.

#### Approach: `background-clip: text` + CSS custom property `--progress`

For the ACTIVE line:
- Instead of individual `.lit` spans, render the ENTIRE line text with `background-clip: text`
- Set `--progress` as a CSS custom property (0% to 100%) based on current time
- CSS creates a gradient: highlighted color up to `--progress%`, dim color after
- This produces the smooth sweep effect without per-character DOM manipulation

For PAST/FUTURE lines:
- Simple opacity styling, no special animation needed

#### Implementation:

**A. Remove the rAF character highlighting loop** (lines 84-110)
No more querying `.karaoke-char[data-start]` every frame.

**B. Change active line detection from 200ms interval to rAF**
Instead of `setInterval(200ms)`, use a rAF loop that:
1. Checks `timeRef.current` against line start times
2. Updates `activeIndex` state only when it actually changes
3. Updates `--progress` CSS variable on the active line element (direct DOM, no React re-render)

**C. For EnhancedLineStatic and SimulatedLineStatic:**
Instead of rendering character spans with `data-start`, render as a single container:

```tsx
// ACTIVE line with gradient sweep
<div
  className="karaoke-text karaoke-sweep"
  style={{ '--progress': `${lineProgress}%` } as React.CSSProperties}
>
  {line.content}
</div>
```

Where `lineProgress` is calculated:
```typescript
const nextStart = lines[activeIndex + 1]?.startMs ?? (line.startMs + 5000);
const duration = nextStart - line.startMs;
const elapsed = currentTimeMs - line.startMs;
const progress = Math.max(0, Math.min(100, (elapsed / duration) * 100));
```

**D. For per-word timing (enhanced/yrc):**
Calculate more precise progress based on word timings:
```typescript
function calcWordProgress(words, timeMs, nextLineStartMs, lineStartMs) {
  let totalChars = 0, litChars = 0;
  for (let w = 0; w < words.length; w++) {
    const word = words[w];
    const wordStart = word.startMillisecond;
    const wordEnd = words[w + 1]?.startMillisecond ?? nextLineStartMs ?? (lineStartMs + 30000);
    totalChars += word.content.length;
    if (timeMs >= wordEnd) {
      litChars += word.content.length;
    } else if (timeMs > wordStart) {
      litChars += word.content.length * ((timeMs - wordStart) / (wordEnd - wordStart));
    }
  }
  return totalChars > 0 ? (litChars / totalChars) * 100 : 0;
}
```

**E. Direct DOM update for --progress (avoid React re-render)**
Use a ref to the active line element and update its style directly in rAF:
```typescript
const activeLineRef = useRef<HTMLDivElement>(null);

// In rAF tick:
if (activeLineRef.current) {
  const progress = calcProgress(activeIndex, timeRef.current, lines);
  activeLineRef.current.style.setProperty('--progress', `${progress}%`);
}
```

**F. Optimized rAF loop**
Only ONE rAF loop handles both:
1. Active line detection (check if index changed)
2. Progress calculation + DOM update for active line

```typescript
useEffect(() => {
  let raf: number;
  const prevIdx = { current: -1 };
  
  const tick = () => {
    const t = timeRef.current;
    
    // Find active line (optimized — start from prevIdx since time only moves forward)
    let idx = -1;
    const startSearch = Math.max(0, prevIdx.current - 1);
    for (let i = startSearch; i < lines.length; i++) {
      if (lines[i].startMs <= t) idx = i;
      else break;
    }
    
    // Update state only on change
    if (idx !== prevIdx.current) {
      prevIdx.current = idx;
      setActiveIndex(idx);
    }
    
    // Update progress via direct DOM
    if (activeLineRef.current && idx >= 0) {
      const nextStart = lines[idx + 1]?.startMs ?? (lines[idx].startMs + 5000);
      const duration = nextStart - lines[idx].startMs;
      const progress = Math.max(0, Math.min(100, ((t - lines[idx].startMs) / duration) * 100));
      activeLineRef.current.style.setProperty('--progress', `${progress}%`);
    }
    
    raf = requestAnimationFrame(tick);
  };
  
  raf = requestAnimationFrame(tick);
  return () => cancelAnimationFrame(raf);
}, [lines]);
```

**G. Scrolling fix**
Change from `scrollIntoView({ block: 'center' })` to manual scroll calculation
that positions the active line at 35% from top (slightly above center, accounting
for the mask-image fade):

```typescript
useEffect(() => {
  if (activeIndex < 0 || !containerRef.current || !activeLineRef.current) return;
  const container = containerRef.current;
  const line = activeLineRef.current;
  const containerH = container.clientHeight;
  const lineTop = line.offsetTop;
  const lineH = line.offsetHeight;
  // Position active line at 35% from top of container
  const targetScroll = lineTop - containerH * 0.35 + lineH / 2;
  container.scrollTo({ top: targetScroll, behavior: 'smooth' });
}, [activeIndex]);
```
